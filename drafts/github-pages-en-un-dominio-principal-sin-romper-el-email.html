GitHub pages en un dominio principal sin romper el email

Date lun 23 junio 2014 Tags github / dns

Aunque estoy muy contento con mi blog lowcost con GitHub y Pelican, no era un consciente de un problema que tenía mi configuración. Y es que entre otras cosas, he perdido el correo del dominio de unos meses...

Como este blog se encuentra en el dominio principal (o apex domain), me pareció razonable en su momento crear una entrada CNAME en las DNS que apuntara de @.pataliebre.net hacia davidcaste.github.io. Lo que no sabía es que en el RFC1034 sección 3.6.2 se comenta explícitamente:

If a CNAME RR is present at a node, no other data should be present; this ensures that the data for a canonical name and its aliases cannot be different.

Por esta razón, si se encuentra una entrada CNAME en el dominio principal se ignoran el resto de entradas, MX incluidas. Lo más triste de todo es que se comenta explícitamente en las sección Setting up a custom domain with GitHub Pages de la documentación de GitHub Pages.

Según Tips for configuring an A record with your DNS provider, hay tres posibles opciones:

  1. Configurar una entrada A en el DNS.
  2. Configurar una entrada ALIAS o ANAME si el proveedor de DNS lo permite.
  3. Utilizar un subdominio como www.

Configurar una entrada A en el DNS

Si no se puede usar una entrada CNAME, parece que la solución obvia es utilizar una entrada A. Pero proveedores como GitHub, Heroku, etc. utilizan sistemas dinámicos de enrutado y de respuesta a congestión
